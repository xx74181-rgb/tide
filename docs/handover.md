**修法**：第 411 行 `note-ov` 加上 `onclick="commitNote(false)"`。

**為什麼用 commitNote(false) 而非新建 closeNote()**：「點外面取消」的語意 = 「跳過備注但保留交易」，剛好就是 `commitNote(withNote=false)` 已經做的事（關 overlay、寫 tx、showUndo、清 pending、回水球頁）。少一個函式名要維護。

**前任的診斷錯了**——transform 算式（`240 - fh/2`）邏輯正確、jsdom 驗證 attribute 都正確套用，純 SVG 渲染也看得到鎖頭。真正的問題是 v15 鎖頭設計本身的不確定性（球內 vs 球外、灰層在底 vs 在頂），所以我們**整個重新設計了月費的視覺架構**。詳見「改動二」。

### Bug 3: proExpandedIdx 跨月未清零

**未動的相同問題**：`sideExpandedIdx`（第 700 行附近）也有跨月不清零的問題，但使用者沒明確要求所以沒改。**建議下一輪一起處理**：

## 改動二：月費的視覺架構整個翻轉

這輪最大的改動。順著問題演化講清楚：

### v15 原始設計（已棄用）
- 月費灰色層從球**底部**往上長
- 球內**鎖頭**浮在灰層中央
- 灰層顏色 `rgba(0,0,0,0.20)`、波浪疊在灰層上面

### v16 設計（本檔現況）
- 月費灰色層從球**頂部**往下長
- 球外**鎖頭 + 「每月需扣繳」文字 + ↓ 箭頭** 永遠顯示（只要有月費）
- 球內不再有鎖頭
- 灰層顏色 `#5E5C58` 實色（不是半透明）
- 灰層**畫在波浪上面**（z-order：波浪 → 灰層）
- 灰層下緣一條 0.5px 細線 `rgba(0,0,0,0.25)` 邊界

### 為什麼一連串地改

每個決定都是為了解前一個問題：

| 決定 | 為什麼 |
|---|---|
| 球外鎖頭 | 球內鎖頭月費小時擠不下；球外可以一直顯示 |
| 灰層在頂 | 「月費 = 鎖住」的隱喻；底部留給「剩餘可花」的水位 |
| 灰層改實色 #5E5C58 | 半透明灰被波浪疊色變淡，使用者「填完月費後第一眼像中灰，記第一筆消費才變深灰」就是這個現象 |
| z-order 翻轉 | 即使月初波浪沖過灰層下緣，灰層仍蓋在波浪上、顏色不變 |
| 弓形幾何 (見改動三) | 月費 1% 應該對應球面 1%，不是 viewBox 高度的 1% |

### 視覺隱喻（給設計師）

```
       🔒 每月需扣繳
            ↓
       ╭─────────╮
       │ ▓▓▓▓▓▓▓ │  ← 深灰 = 月費（鎖住，從頂往下）
       │─────────│  ← 細線 = 月費 / 主動消費 邊界
       │░░░░░░░░░│  ← 米色 = 主動消費已花掉
       │═════════│  ← 波浪面 = 剩餘水位
       │▒▒▒▒▒▒▒▒▒│  ← 藍色波浪 = 剩餘可花
       │▒▒▒▒▒▒▒▒▒│
       ╰─────────╯
```

三段語意：
- **頂深灰**：被鎖在 Tide 之外、自動處理的月費
- **中米色**：主動消費已花的金額（球底色露出）
- **底藍色**：本月還可以花的水位

「鎖頭在球外」的設計：鎖頭代表「**承諾**」，不在水球的「狀態空間」內；球內任何元素都是動態狀態（會隨時間變化），鎖頭是恆定的「告知」。

### 為什麼鎖頭是 SVG 不是 emoji

v15 用 emoji 很少。鎖頭用 SVG（11×13px）讓它和球內元素的視覺語彙一致，且能透過 `currentColor` 跟著 `--ink-soft` 變色。

## 改動三：弓形幾何（這輪最大的技術坑）

### 問題
v15 原版用 `fh = ratio × 240` 算灰層高度（線性）。但球體是個**圓**，圓頂端因為弧度，**線性 1% 的 fh 視覺上佔不到圓面積的 1%**——例如 ratio=3.76% 線性 fh=9px，但 9px 高的弓形面積只佔球面 0.6%，使用者看「月費佔球面 0.6%」會覺得跟側邊欄寫的「月費 3.76%」對不上。

### 修法
新函式 `ratioToCapHeight(ratio, r)` 解方程：

```
弓形面積 / 圓面積 = ratio
解 θ from: θ - sin(θ) = 2π × ratio
然後 cap_height = r × (1 - cos(θ/2))
```

### 子坑：Newton 法不穩

我第一版用 Newton-Raphson 迭代解 θ。**對某些小 ratio（0.5% / 2% / 5%）會發散到完全錯誤的值**（差 10-50 倍）。例如使用者預算 $50,000、月費 $748（1.5%），灰層算出 fh=202px 佔了 85% 球面，看起來像「全黑球」。

**修法**：改用二分搜尋（bisection）。40 次迭代達到 1e-12 精度，每幀 < 0.01ms，效能完全不是問題，但**保證收斂**。

```js
function ratioToCapHeight(ratio, r) {
  ratio = Math.max(0, Math.min(1, ratio));
  if (ratio === 0) return 0;
  if (ratio === 1) return 2*r;
  const target = 2 * Math.PI * ratio;
  let lo = 0, hi = 2 * Math.PI;
  for (let i = 0; i < 40; i++) {
    const mid = (lo + hi) / 2;
    if (mid - Math.sin(mid) < target) lo = mid;
    else hi = mid;
  }
  return r * (1 - Math.cos(lo / 2));
}
```

**給未來改演算法的人**：對非線性方程，先用 bisection 確保正確，效能不夠再考慮 Newton（且要在所有可能輸入區間驗證收斂）。Newton 看起來收斂 12 次就好，但**死亡區可能藏在你沒測到的 ratio**。

### 額外的 `+5` offset

`fh = ratioToCapHeight(ratio, 115) + 5` 多了一個 `+5`。原因：rect 從 y=0 開始畫，但球頂在 y=5（cx=120, cy=120, r=115，球頂 y = 120 - 115 = 5）。viewBox 是 240×240 但球佔 230×230，上下各 5px 邊距。`+5` 是讓 rect 的可見部分（被 clip 圓裁切後）正好等於弓形高度。

## 改動四：商業模式（試用 → 永久買斷）

### 演化路徑

> 這一段對未來修改商業模式很重要——我們試過了訂閱制，然後改回買斷，這裡記錄為什麼。

**v15 原版**：NT$100 一次買斷終身解鎖月底回顧。

**中間有過一輪**「NT$149 / 28 天訂閱」的版本（含寬限期、續訂機制、5 種狀態）——做完後使用者決定**不要訂閱、要回到買斷**，但保留「28 天試用」概念。

**v16 現在的模型**：
- 新使用者第一次開 app 自動進入 **28 天試用**（圓餅圖可看）
- 試用最後 5 天，pro-sheet 頂部顯示倒數橫條 + 「永久解鎖」按鈕
- 試用過期 → 圓餅圖鎖回購買畫面
- 一次付 $149 → `proOwned = true`，永久解鎖

### 4 個狀態

| 狀態 key | 條件 | 設定頁副標 | pro-sheet |
|---|---|---|---|
| `trial` | 未買 + 試用剩 > 5 天 | `試用期還剩 X 天` | 直接看圓餅圖，無橫條 |
| `trial-ending` | 未買 + 試用剩 1~5 天 | `試用還剩 X 天` | 圓餅圖 + 灰色倒數橫條（CTA: 永久解鎖） |
| `expired` | 未買 + 試用過期 | `永久解鎖 NT$149` | 購買畫面（CTA: 永久解鎖 NT$149） |
| `purchased` | 已付 | `已解鎖` | 直接看圓餅圖，無橫條 |

### 資料模型
```js
state.proOwned: boolean       // 是否已買斷
state.trialStartAt: number    // 試用開始時間戳 (Date.now())
// 舊欄位 state.pro 還在（為了 hydrate 兼容），實際邏輯只看 proOwned
```

### 核心函式
- `getProStatus()` → `{key, remainDays}` — 主要的狀態判斷
- `proCanView()` → boolean — `key in {trial, trial-ending, purchased}`
- `buyPro()` — 寫入 `proOwned = true`，**沒有金流**（虛擬建置）
- `debugSetTrialDays(d)` — prototype 用，跳到指定試用剩餘天數

### 設定頁有個 Debug 區
設定頁底部現在有一塊**橘色虛線框**「測試用 · Pro 狀態跳期」，5 顆按鈕：
- 剛開始試用 28 天
- 試用剩 5 天
- 試用剩 1 天
- 試用已過期
- 已永久解鎖

**這是 prototype 專用**。代碼中用 `/* === PROTOTYPE DEBUG: ... 上線前移除 === */` 註解標記，正式上線前要刪除整段。

## 改動五：welcome 頁也加月費灰層

v15 原版 welcome 頁的 SVG 完全沒有月費灰層（只有兩條波浪），跟主水球頁視覺不一致——onboarding 結束時看到的水球**沒有鎖頭月費感**，進主水球頁突然冒出灰層，體驗斷裂。

**修法**：welcome SVG 加 `w-fixed-layer` 和 `w-fixed-edge`，`showWelcome()` 內用 `fixedTotal / budget`（**月費總額**，不是已扣金額）算高度。

差別在哪裡：
- **主水球頁**：灰層高度 = `fixedSpent / budget`（已扣月費）→ 月初灰層小、月底灰層大
- **welcome 頁**：灰層高度 = `fixedTotal / budget`（全月月費總承諾）→ 永遠顯示完整月費佔比

設計意圖：welcome 是「未來會自動扣」的承諾，要呈現整月鎖定總量；主水球頁是「目前已被扣的」現況。

## 已知未解問題 / 留給下一輪

### 1. 月費和主動消費共用同一水位（重大未解問題）

> 這是 v15 handover 第 312 行就提到的問題。v16 沒解，視覺翻轉只是繞過了它，**根本解仍未做**。

**現象**：水位 R = 1 - (所有 spent) / budget，**月費也算進 spent**。月初剛扣完月費就看到水位降一截，跟 Tide「月費自動處理、你只管彈性空間」的敘事矛盾。

**真解**（路徑 B）：水位只算主動消費，月費完全不進水位計算。
```js
R = 1 - manualSpent / (budget - fixedTotal)
```

**影響範圍**：
- 水位算法（`getStateInfo`）
- 所有文案的觸發條件（`pickCopy` 的 key 對應的水位區間）
- side-stats 的數字呈現
- welcome 頁的水位（已經是「扣完月費後」的水位 = 隱含已做了分離，但只在這一頁）

這是 v17 的主菜，不是小改。
